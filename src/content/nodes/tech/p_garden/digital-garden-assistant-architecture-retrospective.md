最近这段时间，我一直在给这个数字花园做一件很诱人的事：给它装上一个真正能对话、能带路、还能慢慢长成“数字分身”的助手。

最开始我的想法其实很直接。既然已经有 `/api/agent`、有 Vercel AI SDK、也有可切换的 provider，那么导航这件事似乎也可以交给模型：用户说一句“带我去数字花园日志”，模型判断意图，调用工具，前端收到 `{ message, action }` 后再驱动星图跃迁。这个设计从纸面上看很完整，甚至很像一个标准的 agent loop。

但真正跑起来之后，我很快发现，问题没有出在“它能不能工作”，而是出在“它为什么总在最关键的时候不稳定”。

## 最先暴露出来的，不是能力不够，而是关键路径太脆

此前我在 `/api/agent` 的导航链路上看到过几类很典型的问题：

- 同一句导航指令，在不同模型上表现不一致。
- 未知目标时，模型会瞎猜、超时、报 `400`，甚至请求悬挂。
- OpenAI-compatible 接口虽然能跑普通聊天，但一旦叠上 tool-calling 和 loop，稳定性就明显下降。
- 模型差异被业务逻辑放大，最后用户感知到的就是“助手有时聪明，有时像坏掉了”。

这类问题最容易让人误判成“是不是 Cloudflare 不支持”“是不是某个模型不行”“是不是平台应该迁走”。但我后来越来越确定，**平台不是主矛盾，模型也不是唯一矛盾，真正的问题在于我把一个高确定性的动作塞进了 LLM 的关键路径**。

导航本质上不是开放式生成任务。它更像一个路由问题：用户说的目标是不是我已有的星球、恒星、专题或别名？如果是，就跳；如果不是，就明确告诉他找不到。这件事和“帮我总结这个网站”“你觉得先看哪部分更合适”完全不是同一类问题。

我之前却试图用同一种 agent 机制同时处理两者。问题就从这里开始了。

## 我真正想修的，不是某个报错，而是架构里的角色分配

在旧方案里，导航要经过这样一条路径：

1. 用户输入到达 `/api/agent`
2. 服务层判断这像是导航意图
3. 模型被要求调用工具
4. 工具再返回结构化结果
5. 前端据此执行跳转

这条链路的最大问题，是我同时给了模型两种互相冲突的要求：

- 如果用户要去某个地方，就必须走工具
- 如果用户说的地方不存在，又要直接说明无法定位

一旦再叠上强制工具调用和多步 loop，模型就很容易进入一种很尴尬的状态：**逻辑上应该拒绝跳转，但系统层又逼着它必须产出一个工具调用。**

这时候模型会出现什么行为，其实已经不是“设计问题”，而是在赌兼容性和运气。有的模型会编一个勉强像样的参数，有的模型会直接报 `Bad Request`，有的则会挂在调用里迟迟不返回。

所以这次我没有继续沿着“换模型、换 provider、换平台”的方向盲目排查，而是先问了一个更根本的问题：

> 数字花园助手里，到底哪些能力应该交给模型，哪些能力应该由本地规则先兜住？

我的答案现在已经很明确了：

> **导航由本地确定性解析层负责，LLM 只负责聊天、解释、总结、推荐与后续开放式能力。**

## 当前稳定版，我把它拆成了四层

为了把这件事说清楚，我现在把数字花园助手理解成四层结构。

### 1. 平台与接口层

这一层基本没动，仍然是：

- Vercel 负责部署
- Astro 负责页面和 API 路由
- `/api/agent` 作为统一入口

我没有在这一轮去迁平台，也没有把 Cloudflare Workers 原生运行时拉进来。因为这不是当前最值钱的动作。平台先保持稳定，能让我把问题收敛在真正需要改的地方。

### 2. 确定性路由层

这是这次改造的核心。

现在所有“带我去某某星球”“打开某个展馆”“去花园看看”这类请求，都会先进入本地 resolver。它会做几件很明确的事：

- 判断这到底是不是导航意图
- 抽取用户真正想去的目标词
- 在本地星图、别名和已知 target 中匹配
- 返回三种结果之一：已解析、明确找不到、根本不是导航

这一步的意义非常大，因为它把“是否能跳转”从模型决策里拿了出来。

换句话说，**现在能不能跳，不再取决于模型临场发挥，而取决于我自己的星图索引是否清楚。**

### 3. 内容理解与工具层

这一层目前还没有完全展开，但边界已经想清楚了。未来很多能力其实都应该放在这里，而不是直接塞给 LLM 生猜：

- 当前星球内容摘要
- 相关文章推荐
- 最近更新读取
- 图谱关系解释
- 阅读路径生成

这些能力的共同点是：它们既不是纯规则，也不该完全依赖模型幻觉。更合理的做法，是先做结构化上下文准备，再把组织语言的部分交给模型。

### 4. LLM 助手层

到了这一层，模型才真正做它擅长的事：

- 回答问题
- 解释概念
- 给出总结
- 提供推荐理由
- 逐步长出更像“数字分身”的表达风格

这就是我现在理解的数字花园助手：**不是一个什么都靠 agent loop 自主完成的系统，而是一个 deterministic router + LLM assistant 的混合架构。**

## 这次改完之后，我给自己定了几条长期红线

如果说这轮修的是架构，那么我更想保留下来的，其实是下面这些原则：

- 导航是确定性 workflow，不是 LLM 自由决策。
- LLM 负责解释和表达，不负责判断“是否能跳转”。
- 高确定性动作优先走本地规则与索引。
- 开放式问题再交给模型。
- 复杂能力按阶段增加，不一开始追求一个全能智能体。
- 先把单助手、多工具、稳定路由做好，再考虑多 agent。
- 先求稳定可控，再追求智能感和自主性。

这些原则看起来像是“保守”，但它们其实是在避免一个很常见的误区：**当我们看到模型能做很多事时，就很容易把所有事情都交给它。**

但真正成熟的系统，往往不是“最自由”的系统，而是边界最清楚的系统。

## 这里开始，我把它当成一份公开的阶段进度表

如果前半篇讲的是“我为什么要改”，那么从这里开始，我更想回答另一个问题：

> 既然方向已经定了，这个数字花园助手接下来到底准备怎么长？

我不想把这部分继续写成一篇只有态度、没有进度的文章，所以决定把后半段直接公开成一份阶段进度表。它不是内部文档原样贴出，而是我自己给这条路线留下的一张公开地图。

状态标记我先约定如下：

- `[x]` 已完成
- `[~]` 已确定方向，或已有基础实现，但还没完全展开
- `[ ]` 还未开始，作为下一阶段 TODO
- `[!]` 关键注意事项或长期风险点

## 当前总览：最脆的那一段已经先拆掉了

就当前版本而言，我最关心的并不是“助手已经多聪明”，而是“它终于不再把导航建立在运气上”。换句话说，**最关键的稳定性修复已经完成，后面的工作才真正有了继续往上叠的基础。**

### 当前已经落地的核心事项

- `[x]` `/api/agent` 仍然保留统一入口与 `{ message, action }` 契约
- `[x]` 导航请求不再依赖模型做最终判断
- `[x]` 本地 `navigation-resolver` 已经接管导航意图识别与目标匹配
- `[x]` 已知目标直接返回 teleport action
- `[x]` 未知目标直接返回无法定位文本
- `[x]` 非导航请求进入独立聊天链路
- `[x]` 聊天链路继续保留 provider 抽象，不绑定单一模型厂商
- `[x]` 导航相关的强制 tool loop 已从关键路径移除
- `[x]` 已补齐回归测试、内容校验与真实 smoke test

这意味着，我现在终于可以把“模型偶发失控导致整个导航体验坏掉”这件事从第一优先级里拿掉了。

## Phase 1：稳定核心链路

这是已经实质推进的一期，也是后面所有能力的地基。

### 目标

先保证助手入口稳定、导航可控、基础对话可用。

### 当前进度

- `[x]` 明确导航动作不再依赖模型决定
- `[x]` 导航解析改为 Resolver-first
- `[x]` 已知目标直接返回 teleport action
- `[x]` 未知目标直接返回无法定位文本
- `[x]` 导航链路不再强依赖 `toolChoice: 'required'`
- `[x]` 导航上的多步 loop 已退出关键路径
- `[x]` 非导航请求进入独立聊天链路
- `[x]` 聊天与导航已拆分为独立文件边界
- `[x]` `/api/agent` 契约回归已完成
- `[x]` 前端终端消费协议无需改动

### 这一阶段的结论

从结果上看，这一步是有效的。之前最容易抖动的就是导航链路，而现在导航已经变成一个确定性的本地流程，模型差异不再直接决定“到底能不能跳”。这不是让系统变笨，而是先把最不该随机的部分稳住。

## Phase 2：内容理解能力

如果说 Phase 1 解决的是“能不能稳定工作”，那么 Phase 2 才开始真正回答“它是不是懂这座花园”。

### 我希望这一阶段完成的事

- `[~]` 构建当前星球上下文加载层
- `[~]` 支持“总结当前星球内容”
- `[~]` 区分当前页面内容与全站内容
- `[ ]` 构建花园总览能力，回答“这个花园主要有哪些内容”
- `[ ]` 面向第一次进入的人提供导览式介绍

### 我心里的重点

这一阶段的关键词不是“更像模型”，而是“更像一个真正读过这座花园的讲解员”。只有当助手开始真正理解当前星球、当前板块、全站结构时，它才会从“会聊天”变成“有内容感”。

- `[~]` 已补充 Phase 2A 的人工验收清单，用来检查 scope-aware answers 是否贴合页面语境

## Phase 3：推荐与发现能力

到了这一阶段，我希望助手不再只是等着被问，而是开始具备一点“带你逛”的能力。

### 当前 TODO

- `[~]` 根据问题推荐相关星球或文章（轻量混合式 MVP）
- `[~]` 为推荐提供理由说明
- `[~]` 返回可直接触发跳转的 action
- `[~]` 支持“你可能还想看”的延伸式推荐
- `[~]` 支持查询最近更新的几个星球
- `[~]` 支持最近新增内容的一句话概述
- `[~]` 支持关系解释与关键节点识别

> 当前 Phase 3 先落的是轻量混合式推荐：同星球、同标签、最近更新三路召回，统一打分后返回推荐与 action；向量召回接口已预留，但还没有接成运行时硬依赖。

### 这一阶段的意义

如果说前一阶段是“懂内容”，这一阶段更像是“懂路径”。用户不一定总是知道自己想去哪里，很多时候他需要的是一条被温和引导出来的探索路线。

## Phase 4：数字分身与访客体验

这是我很期待，但也会明显更谨慎的一期。

### 当前 TODO

- `[ ]` 让回答风格更接近我本人的表达方式
- `[ ]` 在输出中体现偏好、语气与思考路径
- `[ ]` 建立人格层与事实层的明确边界
- `[ ]` 支持新访客导览
- `[ ]` 支持入门路线推荐
- `[ ]` 区分新访客 / 老访客 / 熟人模式

### 这一阶段的风险点

- `[!]` 回答越像“数字分身”，越要防止事实漂移
- `[!]` 风格可以增强，但不能靠瞎编来制造“像本人”的错觉
- `[!]` 任何人格化表达都应该建立在公开内容和结构化上下文之上

我确实希望这个助手未来更像“我自己”，但这件事一定不能靠放大幻觉来实现。

## Phase 5：更成熟但仍然可控的花园助手

再往后，我才会考虑那些更接近 agent 的能力，但前提仍然是可控。

### 当前 TODO

- `[ ]` 阅读路径生成
- `[ ]` 主题漫游
- `[ ]` 时间线回顾
- `[ ]` 关系解释
- `[ ]` 关键节点识别
- `[ ]` 更成熟的主题式探索体验

后续补充：内容索引真相源、效果评测、运行观测、内容更新刷新机制。

### 我的态度

这些能力并不是不会做，而是我不打算为了“看起来更高级”而提前把系统做成一个又厚又重的 agent 工程。它们应该长在已经清楚分层的架构上，而不是反过来把现在好不容易理顺的边界重新搅乱。

## 我现在最想避免的，是重蹈旧路线

做完这轮复盘以后，我反而比之前更清楚哪些事情暂时不要做：

- 不要重新把导航关键路径塞回 LLM
- 不要再把“是否能跳转”设计成 prompt 决策问题
- 不要为了“更像智能体”就过早引入多智能体
- 不要为了未来可能会用到的能力，先把系统做得过于复杂
- 不要让 provider 切换反过来污染业务层逻辑
- 不要在边界还没稳的时候就引入复杂 workflow engine、memory 系统或运行时 fallback

这些“不做”的决定，其实和“要做什么”一样重要。

因为数字花园助手真正需要的，不是一次性长成一个全能系统，而是沿着正确方向逐步生长。

## 这份进度表对我来说，不只是 TODO，更像一张地图

我把这篇文章写下来，不只是为了记录“这次改了什么”，也是为了给之后的开发留一张地图。它一半是复盘，一半是进度表；前半段用来解释我为什么这样改，后半段则用来提醒我接下来应该沿着哪条路线继续做。

以后每推进一步，我都可以回头问自己几个问题：

- 我现在做的，是确定性路由的问题，还是开放式解释的问题？
- 这个能力应该长在 resolver 层、内容理解层，还是 LLM 层？
- 我是在增强助手，还是在重新制造不稳定？

如果这些问题还能被清楚回答，那说明这条路还没有走偏。

## 最后一句话

当前数字花园助手的正确发展路径，不是让模型决定一切，而是：

> **先用确定性规则稳住导航与结构，再让 LLM 逐步承担解释、总结、推荐和数字分身式对话。**

对我来说，这不是一次保守回退，而是一次更清醒的架构前进。
